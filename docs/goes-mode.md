# GOES observation mode

The telescope supports two observation modes, selected at boot on the
hardware service:

| | Hydrogen line (default) | GOES |
|---|---|---|
| LNA | Sawbird+ H1 | Sawbird+ GOES |
| Carrier | 1420.4 MHz | 1694.1 MHz |
| Config | `[sdr]` | `[goes]` |
| DSP / decode | `rt_hardware.sdr_pipeline` (GNU Radio FFT) | goestools (`goesrecv` + `goesproc`) |
| Service | `SpectrumService` → `/ws/spectrum` | `GoesService` → `/ws/goes` |
| Frontend | `SpectrumPanel`, baseline wizard, guided observation | `GoesConnectPanel`, `GoesDataExplorer` |

## Switching modes

1. Swap the LNA on the feed (both can be powered through the SDR's bias tee
   via `lna_bias_tee_enabled` in the respective section).
2. In `hardware/config.toml` set:

   ```toml
   [observation]
   mode = "goes"   # or "hydrogen_line"
   ```

3. Restart `rt-hardware`. The platform needs no config change and no restart
   — it proxies both surfaces unconditionally and the frontend picks its
   panel set from `GET /api/observation`.

The modes are deliberately segregated: each has its own config section,
subprocess chain, service, routes, WS bridge, and frontend components. Only
the selected mode's service is instantiated (the single SDR can't be shared).

## GOES receive chain (goestools)

Demodulation and decoding are delegated to
[goestools](https://github.com/pietern/goestools) — the community-standard,
battle-tested GOES receiver — rather than maintained in this repo.
`GoesService` is supervision + glue:

```
goesrecv  (subprocess; config generated by rt-hardware)
   SDR → AGC → RRC → clock recovery → Costas → Viterbi → deframe → RS(255,223)
     ├─ demod stats    nanomsg JSON   {gain, frequency, omega}
     ├─ decoder stats  nanomsg JSON   {viterbi_errors, reed_solomon_errors, ok, ...} per frame
     ├─ symbols        nanomsg int8   post-clock-recovery I/Q
     └─ VCDUs          nanomsg 892 B  error-corrected frames
                          │
goesproc  (subprocess; cwd = product store) ──────────► image / text files on disk
                          │
GoesService (asyncio)
   ├─ consumes the four publishers over a minimal nanomsg-SUB client
   │  (rt_hardware/goes/nanomsg.py — SP handshake + length-prefixed frames)
   ├─ SNR estimate + constellation from the symbol stream,
   │  per-virtual-channel counters from VCDU headers
   ├─ merges everything into /ws/goes status frames
   └─ indexes goesproc's output tree into the ProductStore (HTTP API)
```

`goesrecv.conf` is regenerated on every spawn from `[goes]`, so the nanomsg
endpoints always match the consumers. The goesproc handler config (crops,
false color, file naming) is auto-discovered from the goestools install,
overridable via `goes.goesproc_config`; a minimal fallback ships with the
package. goesproc runs with the product store as its working directory, so
handler-relative output paths land where `/api/goes/products` serves files.

goestools is built from source in the hardware Docker image; on bare metal
install it per its README. When the binaries are missing the service reports
`mode = "unavailable"` with an actionable fault detail.

## Simulate mode

```toml
[observation]
mode = "goes"

[goes]
simulate = true
```

No SDR or goestools required. The simulator produces the same observation
shapes the nanomsg consumers would (demod/decoder stats, symbol batches,
VCDU headers) and writes demo products (synthetic full-disk imagery, NWS
bulletins) into the same directory tree goesproc would — status frames,
product indexing and the entire frontend run unchanged. Signal acquisition
follows the dish: SNR rises as the mount closes on the target satellite's
look angles, so the slew → peak → lock workflow is faithfully demoable.

## Surfaces

Hardware (trusted network): `GET /api/observation`, `GET /api/goes/status`,
`POST /api/goes/reconnect`, `GET /api/goes/products`,
`GET /api/goes/products/{id}/file`, `WS /ws/goes`.

Platform (public, queue-gated): same paths are proxied; live reads and streams
need the active queue session, and mutations such as reconnect need control.
`/api/observation` degrades to `{"mode": "hydrogen_line", "degraded": true}`
when the gateway is down.

## UI flow

1. **Connect panel** (right column, replaces the hydrogen-line spectrum
   panel): pick a satellite from the catalog, *Slew to satellite*, then peak
   the SNR meter — the acquisition stepper walks Searching → Signal → Frame
   lock → Data. A rolling SNR history and live constellation show what the
   demodulator sees.
2. **Data explorer** (full-width below the sky map, appears at frame lock):
   link statistics (data rate, frame drops, Viterbi/Reed-Solomon correction
   rates), per-virtual-channel activity, and a gallery of decoded products
   with a lightbox viewer and download links.
